# Onboarding — 99names

Dit document is bedoeld voor mensen die **met dit project willen werken**, eraan willen bijdragen, of het willen beoordelen.

Lees dit **volledig** vóór je iets doet.

---

## 1. Wat je hier aantreft

Dit project is een **gestructureerde kennis- en verwerkingsinfrastructuur**.
Het draait om:
- zorgvuldige omgang met islamitische brondata
- herleidbare verwerking
- expliciete besluitvorming
- weerstand tegen improvisatie en scope creep

Dit is geen:
- experimenteel speeltje
- snelle ML-playground
- losse verzameling scripts

---

## 2. Verwachtingen

Als je hier werkt, wordt van je verwacht dat je:

- documentatie leest vóór je handelt
- aannames expliciet maakt
- geen shortcuts neemt
- werkt binnen je rol
- accepteert dat niet alles snel gaat

Als je energie krijgt van “even snel fixen”, zit je hier verkeerd.

---

## 3. Begin hier (verplicht leespad)

Lees in deze volgorde:

1. `PROJECT_OVERVIEW.md`  
   → wat dit project is en niet is  

2. `DECISIONS.md`  
   → welke keuzes al vastliggen  

3. `ROLES.md`  
   → welke rollen bestaan  

4. Jouw specifieke rol in `/roles/`  
   → wat jij wel en niet doet  

5. `WORKFLOW.md`  
   → hoe werk zich door het project beweegt  

Pas daarna:
- code
- scripts
- tooling
- infrastructuur

---

## 4. Rollen zijn leidend

Je handelt **altijd vanuit één rol**.

- Heb je geen rol? Dan handel je niet.
- Wil je iets doen buiten je rol? Dan escaleer je.
- Vind je een rol onhandig? Dan documenteer je dat.

Rollen voorkomen verwarring. Ze zijn geen formaliteit.

---

## 5. Hoe werk ontstaat

Werk begint niet met implementatie, maar met:
- een vraag
- een taak
- een expliciete opdracht

Uitvoering zonder context is ongeldig.

Elke significante uitvoering leidt tot:
- een rapport in `/reports/`
- volgens `REPORT_TEMPLATE.md`

---

## 6. Reviews en kritiek

Kritiek wordt niet informeel afgehandeld.

Er bestaan expliciete review-vormen:
- interne review
- code review
- security review
- pentest
- red-team analyse

Zie `/reviews/` voor kaders.

Reviews:
- implementeren niets
- beslissen niets
- leveren input

---

## 7. Besluiten nemen

Structurele keuzes worden **alleen** genomen door de Project Owner.

Als iets:
- architectuur raakt
- scope wijzigt
- risico’s introduceert

dan vereist dat een vastgelegde beslissing in `DECISIONS.md`.

“Het leek logisch” is geen besluit.

---

## 8. Communicatie

Dit project vertrouwt niet op mondelinge afspraken.

- alles wat telt wordt geschreven
- discussie leidt tot documentatie
- onduidelijkheid leidt tot escalatie

Stilte betekent: geen besluit.

---

## 9. Wanneer je beter kunt stoppen

Dit project is waarschijnlijk **niet** geschikt voor je als:

- je vrijheid zoekt om te experimenteren
- je snelheid boven zorgvuldigheid stelt
- je moeite hebt met grenzen
- je allergisch bent voor documentatie
- je “gewoon wilt bouwen”

Dat is geen oordeel. Dat is afbakening.

---

## 10. Slot

Dit project is zo ingericht dat het:
- traag mag zijn
- kritiek verdraagt
- overdraagbaar blijft
- niet afhankelijk is van heldendom

Als je na het lezen nog steeds wilt bijdragen:
- kies je rol
- handel binnen de kaders
- schrijf alles wat telt op

Welkom.
